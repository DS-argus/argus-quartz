> [!abstract]+ TL;DR
> - macOS에서 `brew install go`로 설치하고 `GOPATH/bin`을 PATH에 추가하면 준비 완료
> - `go run`은 테스트용 임시 실행, `go build`는 배포용 바이너리 생성
> - `go install`은 `pipx install`에 가까운 개념이고, `go get`이 `pip install`에 해당
> - Go는 `go fmt`가 사실상 강제라 포매팅 논쟁이 없음
> - 프로젝트 시작 시 `go mod init`으로 모듈 초기화 필수

> [!note]+ 이 시리즈에 대하여
> Jon Bodner의 *Learning Go*를 읽고 정리한 Go 입문 시리즈다. Python에 익숙한 개발자가 Go를 처음 만났을 때를 기준으로, 두 언어를 비교해 가며 개념을 풀어낸다.

### 1. 설치와 환경 변수

macOS 기준으로 Homebrew를 사용하면 간단하다.

```bash
brew install go
go version
# go version go1.24.2 darwin/arm64
```

Go를 설치하면 두 가지 경로가 중요하다.

- `GOROOT` : Go 자체가 설치된 경로 (보통 건드릴 일 없음)
- `GOPATH` : Go가 외부 패키지와 바이너리를 저장하는 작업 디렉토리

`GOPATH`의 기본값은 `~/go`이고, `go install`로 설치한 바이너리가 `$GOPATH/bin`에 들어간다. 이 경로를 PATH에 추가해야 설치한 도구를 바로 실행할 수 있다.

```bash
# ~/.zshrc 또는 ~/.bashrc에 추가
export GOPATH="$HOME/go"
export PATH="$GOPATH/bin:$PATH"
```

현재 설정을 확인하려면 `go env`를 쓰면 된다.

```bash
go env          # 전체 환경 변수 출력
go env GOPATH   # 특정 변수만 확인
go env GOROOT
```

> [!tip]+ Python과의 차이
> Python은 `sys.prefix`, `site-packages` 같은 경로가 가상환경마다 달라지지만, Go는 `GOPATH` 하나로 글로벌하게 관리된다. 프로젝트별 의존성 격리는 Go 모듈(`go.mod`)이 담당한다.

### 2. 실행 방식 — `go run` vs `go build`

Python은 인터프리터가 소스 코드를 직접 읽어서 실행한다. Go는 컴파일 언어라서 반드시 바이너리를 만든 뒤 실행하는 구조다. 다만 `go run`이 이 과정을 한 번에 처리해 준다.

#### `go run` — 테스트용 실행

```bash
go run main.go
```

내부적으로 임시 디렉토리에 바이너리를 만들고, 실행하고, 끝나면 삭제한다. Python에서 `python main.py` 치는 것과 체감이 비슷하지만, 실제로는 컴파일이 일어나고 있다.

#### `go build` — 바이너리 생성

```bash
go build -o myapp main.go
./myapp
```

`-o` 옵션으로 출력 파일명을 지정한다. 생성된 바이너리는 Go가 설치되지 않은 환경에서도 실행 가능하다. Python으로 치면 PyInstaller로 패키징한 것과 비슷한 결과물이지만, Go는 이게 기본 동작이다.

```bash
# 정리
go run main.go           # 컴파일 + 실행 + 임시파일 삭제 (개발 중 테스트)
go build -o myapp main.go  # 컴파일 + 바이너리 생성 (배포용)
```

### 3. 패키지 설치 — `go install`과 `go get`

Go에서 뭔가를 "설치"하는 명령어는 두 가지다. 이름이 비슷해서 처음에 헷갈리지만, 용도가 완전히 다르다.

#### `go install` — CLI 도구 설치

Go로 만들어진 실행 파일(CLI 도구)을 내 시스템에 설치하는 명령어다. 소스 코드를 받아서 컴파일한 뒤, 완성된 바이너리를 `$GOPATH/bin`에 넣어준다.

```bash
# golangci-lint 설치 (린팅 도구)
go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest

# 설치 확인
which golangci-lint
# /Users/you/go/bin/golangci-lint
```

`@latest`는 최신 버전을 의미하고, `@v1.62.0`처럼 특정 버전도 지정할 수 있다. 결과물은 단일 바이너리 파일이라서 설치 후에는 Go가 없어도 실행 가능하다.

프로젝트 의존성과는 무관하다. `go.mod`에 아무것도 추가되지 않는다.

#### `go get` — 프로젝트 라이브러리 추가

현재 프로젝트에서 import해서 사용할 라이브러리를 추가하는 명령어다. `go.mod`와 `go.sum`에 의존성이 기록된다.

```bash
# 프로젝트에서 사용할 웹 프레임워크 추가
go get github.com/gin-gonic/gin

# 특정 버전 지정
go get github.com/gin-gonic/gin@v1.10.0
```

실행해도 바이너리가 생기지 않는다. 코드에서 `import "github.com/gin-gonic/gin"`으로 가져다 쓰는 용도다.

#### Python 대응 정리

| Go | Python | 용도 |
|---|---|---|
| `go install <패키지>@latest` | `pipx install` / `uv tool install` | CLI 도구를 시스템에 설치 |
| `go get <패키지>` | `pip install` / `uv add` | 프로젝트 의존성으로 라이브러리 추가 |
| `go mod tidy` | `pip freeze` / `uv sync` | 의존성 정리 및 동기화 |

> [!tip]+ 핵심 차이
> Python에서는 `pip install`이 도구든 라이브러리든 가리지 않고 설치한다. `pipx`가 나온 이유도 이 구분이 모호해서다. Go는 처음부터 `go install`(도구)과 `go get`(라이브러리)으로 명령어가 분리되어 있다.

### 4. 코드 포매팅과 린팅

Python에서는 black, ruff, isort, flake8 등 여러 도구를 조합해서 쓴다. Go는 포매팅 도구가 언어에 내장되어 있고, 사실상 강제다.

#### `go fmt` — 기본 포매터

```bash
go fmt ./...
```

`./...`는 현재 디렉토리 하위의 모든 Go 파일을 대상으로 한다. 탭 vs 스페이스, 중괄호 위치 같은 스타일 논쟁이 Go에서는 존재하지 않는다. `go fmt`가 정한 스타일이 곧 표준이다.

#### `goimports` — import 자동 정리

```bash
go install golang.org/x/tools/cmd/goimports@latest
goimports -w .
```

`go fmt`의 기능에 더해서 사용하지 않는 import 제거, 빠진 import 자동 추가를 해준다. Python의 isort + autoflake에 해당한다.

#### `go vet` — 정적 분석

```bash
go vet ./...
```

컴파일은 통과하지만 버그 가능성이 높은 코드를 검출한다. 예를 들어 `fmt.Printf`에서 포맷 문자열과 인자 개수가 안 맞는 경우를 잡아준다.

#### `golangci-lint` — 통합 린팅 도구

```bash
go install github.com/golangci/golangci-lint/cmd/golangci-lint@latest
golangci-lint run
```

`go vet`을 포함해서 수십 개의 린터를 한 번에 돌려주는 도구다. Python의 ruff가 여러 린터를 통합한 것과 비슷한 포지션이다.

> [!tip]+ Python vs Go 포매팅 비교
> - Python: black/ruff는 선택이고, 팀마다 설정이 다를 수 있다
> - Go: `go fmt`가 유일한 표준이다. 설정 파일도 없고, 옵션도 없다
> - 결과적으로 Go 프로젝트는 어디를 가든 코드 스타일이 동일하다

### 5. 모듈 초기화 (`go mod`)

Go 1.16부터 모듈 모드가 기본이다. 새 프로젝트를 시작하면 `go mod init`을 먼저 실행해야 한다.

```bash
mkdir myproject && cd myproject
go mod init github.com/username/myproject
```

이 명령어가 `go.mod` 파일을 생성한다. Python의 `uv init`이나 `pyproject.toml` 생성에 해당한다.

```
module github.com/username/myproject

go 1.24.2
```

이후 `go get`으로 패키지를 추가하면 `go.mod`에 의존성이 기록되고, `go.sum`에 체크섬이 저장된다.

```bash
go get github.com/gin-gonic/gin
```

> [!note]+ 모듈 관련 참고
> - `go mod tidy` : 사용하지 않는 의존성 제거, 빠진 의존성 추가
> - `go mod download` : `go.mod` 기반으로 의존성 다운로드 (`uv sync`와 유사)
