### 도메인 네임과 네임 서버
- IP 주소만으로 모든 호스트를 식별하는데는 몇 가지 불편함이 있음
	- 모든 호스트 IP 주소 기억 어려움
	- 호스트 IP는 바뀔 수 있음

##### 도메인 네임 : 호스트의 IP 주소와 대응되는 문자열 형태의 특정 정보
- www.naver.com, ...
- IP 주소가 전화번호라면, 도메인 네임은 사용자 이름

##### 네임 서버 : 도메인 네임과 IP 주소를 관리하는 서버
- 그 중 도메인 네임을 관리하는 네임 서버는 DNS 서버
- 도메인 네임을 네임 서버에 물어보면 IP 주소 알려줌
	- resolving : IP 주소를 모르는 상태에서 도메인 네임에 대응되는 IP 주소를 알아내는 과정

##### 도메인 네임의 구조
- 도메인 네임의 뒤에서부터 '.'을 기준으로 계층적으로 분류하며 일반적으로 3~5 단계로 구성
- 전체 주소 도메인 네임 (FQDN, Fully-Qualified  Domain Name) : www.naver.com
	- 루트 도메인 : .  -> 보통 생략
	- 최상위 도메인 (Top Level Domain) : com
	- 2단계 도메인 : naver
	- 3단계 도메인 : www
- 호스트 네임 : FQDN의 첫 번째 부분 (www)
	- FQDN 전체 의미하기도 함

##### DNS (Domain Name System) : 계층적이고 분산된 도메인 네임에 대한 관리 체계 or 호스트가 DNS를 이용할 수 있도록 하는 애플리케이션 계층 프로토콜
- 계층적인 도메인 네임을 효율적으로 관리하기 위해 네임 서버도 계층적 형태를 가짐
- 또한 전 세계 여러 군데 위치해 있음


> [!info] 서브 도메인
> 다른 도메인이 포함된 도메인을 의미
> - 예시 : google.com의 서브 도메인
> 	- mail.google.com
> 	- www.google.com
> 	- scholar.google.com
> 	- drive.google.com
> 	- ...

### 계층적 네임 서버
##### 다양한 네임 서버 종류
###### 1. 로컬 네임 서버
- 클라이언트가 가장 먼저 찾는 네임 서버로 우선 클라인트가 접속하기 위한 주소 필요
	- [[1. 네트워크 거시적으로 살펴보기#^80a17b|ISP]]에서 로컬 네임 서버 주소 할당
	- 공개 DNS 서버 이용
		- 8.8.8.8, 8.8.4.4 (구글)
		- 1.1.1.1 (클라우드 플레어)
- 대응되는 IP 주소를 알면 바로 반환
- 만약 모른다면 루트 네임 서버에 질문

###### 2. 루트 네임 서버 : www.example.com 주소? -> com 관리하는 네임 서버 IP 주소 반환
- 로컬 네임 서버가 도메인 네임에 대응되는 IP 주소를 모를 때 질의하는 서버
- 루트 도메인을 관장하는 네임 서버
- 'com'과 같은 최상위 도메인 (TLD) 네임 서버의 IP 주소 반환

###### 3. TLD 네임 서버 : TLD를 관리하는 네임 서버 : www.example.com 주소? -> example.com 관리하는 네임 서버 IP 주소 반환
- 하위 도메인 네임 서버에 대한 IP 주소를 알려줌
- 계속 하위 도메인에 대한 네임서버를 재귀적으로 찾아감

###### 4. 책임 네임 서버
- 자신이 관리하는 도메인 영역에 속한 질의가 오면 곧바로 대답
- 로컬 네임 서버가 마지막으로 질의하는 네임 서버

> 로컬 네임 서버 -> TLD 네임 서버 -> • • • -> 책임 네임 서버  순서로 질의해서 원하는 IP 주소 알아냄

##### Resolving 과정
###### 재귀적 질의 : 가장 단순한 방식
- 질문 : 클라이언트 -> 로컬 네임 서버 -> 루트 네임 서버 -> TLD 네임 서버 -> ...
- 응답은 질문의 역순

###### 반복적 질의
- 로컬 네임 서버가 중개 역할 
	- 루트 네임 서버로부터 TLD 네임 서버 IP 받음
	- TLD 네임 서버로부터 하위 네임 서버 IP 받음
	- ...
	- 알아내면 클라이언트에 전달

> [!tip] DNS 캐시
> - 재귀적, 반복적 질의 모두 시간이 오래 걸리고 네트워크 상 메시지 수가 지나치게 늘어날 수 있음  
> - 따라서 캐시를 이용해서 네임 서버들이 기존에 응답받은 결과를 임시 저장했다고 같은 질의에 재활용
> - TTL 만큼에 해당하는 시간 동안만 캐시해놓음

### 자원[^1]을 식별하는 URI
- URI (Uniform Resource Identifier) : 자원을 식별할 수 있는 정보
	- 네트워크상에서 자원을 주고 받으려면 먼저 식별을 해야함
	- 세부적 방식으로 URL, URN 나뉨

##### URL (Uniform Resource Locator) : 위치를 이용한 자원 식별 방법
- 더 많이 사용
- 형식  
	![[13. DNS와 자원 - 2025-06-04 - 11-49-07.png|786x84]]
###### 1. scheme : 자원에 접근하는 방법
- 사용할 프로토콜 명시
	- HTTP 사용 -> http://
	- HTTPS 사용 -> https://
###### 2. authority : 호스트를 특정할 수 있는 정보
- IP 주소 혹은 도메인 네임
- 포트 번호까지 추가 가능

###### 3. path : 자원이 위치한 경로
- / 기준으로 계층적으로 표현

###### 4. query : HTTP에 요청할 구체적인 요구사항
- 쿼리 문자열, 쿼리 파라미터 작성
- ?로 시작되는 <key=value>형태로 &로 여러 쿼리 문자열 연결 가능

###### 5. fragment : 자원의 한 조각을 가리키는 정보
- `https://datatracker.ietf.org/doc/html/rfc3986` : 해당 html 파일 첫 부분 열림
- `https://datatracker.ietf.org/doc/html/rfc3986#section-1.1.2` : 해당 html 파일의 1.1.2 섹션 열림

##### URN (Uniform Resource Name) : 이름을 이용한 자원 식별 방법
- 자원의 위치가 변하면 URL로 식별 불가능
- 자원에 고유한 이름을 붙여서 식별
	- `urn:isbn:045145023` : ISBN이 045145023 인 도서를 나타내는 URN
	- `urn:ietf:rtc:2648` : IETF 공식 문서 중 RFC 2648을 나타내는 URN
- 일반 웹브라우저상에서 URN만으로 원하는 페이지로 이동 불가능
	- 수동으로 URL로 변경하거나 URN 해석 서비스를 이용하거나 해야함

 > [!info] DNS 자원 레코드
  > DNS 자원 레코드(Resource Record)는 네임 서버가 저장하고 관리하는 도메인 관련 정보로 도메인 이름을 구입한 뒤 웹 사이트, 메일 서버, 외부 서비스 등을 연결할 때 DNS 레코드를 등록한다.
  >
  > 예를 들어 `example.com`이라는 도메인을 구입했고, 이 도메인으로 접속한 사용자를 서버 `1.2.3.4`로 보내고 싶다면 네임 서버에 다음과 같은 레코드를 등록한
  다.
  >
  > - 호스트 이름: `example.com.`
  > - 값: `1.2.3.4`
  > - TTL: `300`
  > - 유형: `A`
  >
  > 이 설정의 의미는 `example.com`을 조회하면 IPv4 주소 `1.2.3.4`를 응답하라는 뜻이다.
  >
  > DNS는 URL 전체를 등록하는 시스템이 아니다. 예를 들어 `https://www.example.com:443/api/users`라는 URL이 있을 때 DNS가 다루는 부분은 `www.example.com`이
  라는 도메인 이름이다. `https`, 포트, 경로(`/api/users`)는 DNS가 아니라 브라우저, 웹 서버, 프록시, 애플리케이션이 처리한다.
  >
  > 주요 레코드 유형은 다음과 같다.
  >
  > - `A`: 도메인 이름을 IPv4 주소에 연결한다. 예를 들어 `example.com A 1.2.3.4`는 `example.com`을 `1.2.3.4`로 해석하게 한다.
  > - `AAAA`: 도메인 이름을 IPv6 주소에 연결한다. IPv6로 서비스를 제공할 때 사용한다.
  > - `CNAME`: 도메인 이름을 다른 도메인 이름의 별칭으로 만든다. 예를 들어 `www.example.com CNAME example.com`은 `www.example.com`이 `example.com`을 가리
  키게 한다.
  > - `NS`: 해당 도메인 또는 하위 도메인의 DNS 정보를 어떤 네임 서버가 관리하는지 지정한다.
  > - `MX`: 해당 도메인의 메일을 어떤 메일 서버가 받을지 지정한다. 예를 들어 `user@example.com`으로 오는 메일을 처리할 서버를 지정할 때 사용한다.
  > - `TXT`: 임의의 텍스트 정보를 저장한다. 도메인 소유권 검증, SPF, DKIM, DMARC 같은 메일 보안 설정에 자주 사용된다.
  >
  > 웹 사이트를 연결할 때 흔한 구성은 다음과 같다.
  >
  > ```text
  > example.com      A      1.2.3.4
  > www.example.com  CNAME  example.com
  > ```
  >
  > 이 경우 사용자가 `www.example.com`으로 접속하면 DNS는 먼저 `www.example.com`이 `example.com`의 별칭임을 확인하고, 다시 `example.com`의 `A` 레코드를 따
  라가 최종 IPv4 주소 `1.2.3.4`를 얻는다. 실제 DNS 응답에서는 `CNAME`과 최종 `A` 레코드가 함께 반환될 수도 있지만, 개념적으로는 별칭을 따라가 최종 주소를
  찾는 과정이다.
  >
  > `TTL(Time To Live)`은 DNS 응답을 캐시에 얼마나 오래 보관할지를 나타낸다. 예를 들어 TTL이 `300`이면 리졸버나 클라이언트가 해당 응답을 최대 300초 동안
  캐시할 수 있다. TTL이 짧으면 변경 사항이 비교적 빨리 반영되지만 DNS 조회가 더 자주 발생하고, TTL이 길면 캐시 효율은 좋아지지만 변경 반영이 늦어질 수 있
  다.
  >
  > 도메인을 실제 서비스에 연결하려면 DNS 등록만으로는 부족하다. DNS는 도메인을 IP 주소로 바꿔줄 뿐이다. 해당 IP의 서버가 외부에서 접근 가능해야 하고,
  HTTP는 보통 `80`, HTTPS는 `443` 포트가 열려 있어야 하며, 웹 서버나 리버스 프록시가 요청을 받아 애플리케이션으로 전달해야 한다. HTTPS를 사용하려면 해당
  도메인에 대한 TLS 인증서도 필요하다.










[^1]: 네트워크상의 메시지를 통해 주고받는 대상 (HTML, 이미지, 동영상, 텍스트 ...)
