### HTTP의 특성
- HTTP (Hypertext Transfer Protocol) : 응용 계층에서 정보를 주고받는 데 사용되는 프로토콜

> [!asidecleanright]+ 복습 : 지금까지 배운 계층별 프로토콜
> 1. 물리 계층, 데이터 링크 계층 -> [[3. 이더넷#이더넷 (Ethernet) 다양한 통신 매체의 규격들과 송수신되는 2. 네트워크 미시적으로 살펴보기 3d6ed0 프레임 의 형태, 프레임을 주고받는 방법 등이 정의된 네트워크 기술|이더넷]]
> 2. 네트워크 계층 -> [[7. LAN을 넘어서는 네트워크 계층#인터넷 프로토콜 (IP) 네트워크 계층의 가장 핵심적인 프로토콜|IP (인터넷 프로토콜)]]
> 3. 전송 계층 -> [[10. 전송 계층 개요-IP의 한계와 포트|TCP, UDP]]
> 4. 응용 계층 -> HTTP
##### 특성 1 : 요청 - 응답 기반 프로토콜
- [[1. 네트워크 거시적으로 살펴보기#노드 정보를 주고받을 수 있는 장치|클라이언트와 서버]]가 서로 HTTP 요청 메시지와 HTTP 응답 메시지를 주고받는 구조로 동작
	- 클라이언트 : HTTP 요청 메시지로 서버의 자원 요청
	- 서버 : HTTP 응답 메시지로 요청받은 자원에 대해 응답
- HTTP 요청 메시지와 HTTP 응답 메시지는 형태가 다름
##### 특성 2 : 미디어 독립적 프로토콜
- HTTP는 자원의 특성을 제어하지 않고 자원과 상호 작용 하는 데 사용할 수 있는 인터페이스만 정의
	- HTML, JPEG, PNG, JSON, XML, PDF 등 모두 가능
- 미디어 타입 (MIME[^1] 타입) : HTTP에서 주고받는 자원의 종류
	- 형태 : `타입/서브타입;매개변수=값` 
		- 타입 : 데이터 유형
		- 서브타입 : 주어진 타입에 대한 세부 유형
		- 매개변수 : 부가적 설명 위해 선택적으로 추가
			- type/html;charset=UTF-8 : HTML 문서이고 사용된 문자는 UTF-8로 인코딩
	- 매우 다양하며 필요에 따라 새롭게 등록 가능
	- 대표적인 타입 - 서브타입
		- text - plain, html, css, javascript
		- image -- png, jpeg, webp, gif
		- video - mp4, ogg, webm
		- audio - midi, wav
		- application - octet-stream, pdf, xml, json, x-www-form-urlencoded
		- multipart - form-data, encrypted
##### 특성 3 : Stateless 프로토콜
- Stateless : 서버가 HTTP 요청을 보낸 클라이언트와 관련된 상태 기억하지 않음 -> 독립적인 요청으로 간주
- 왜 stateless?
	- 많은 클라이언트와 동시 상호 작용하는데 상태 기억하면 부담
	- 여러 서버로 구성되어 있을 경우 서버 간 상태 정보 공유는 복잡
	- 필요한 경우 언제든 서버 추가 가능해서 확장성 높음
	- 문제 생긴 서버 쉽게 교체 가능해서 견고성 높음
- 상태를 유지하지 않는 특성을 보완하기 위한 몇 가지 방법 있음
##### 특성 4 : 지속 연결 프로토콜
- 과거 HTML (HTML 1.0 이하)는 비연결형 프로토콜
	- 3-way handshake로 TCP 연결 수립 -> 요청, 응답 -> 연결 종료
	- 추가적인 요청, 응답하려면 TCP 재연결
- 최근 HTML (HTML 1.1 이상) : 지속 연결 프로토콜 (keep-alive)
	- 하나의 TCP 연결상에서 여러 요청, 응답 주고받을 수 있음

##### 시작 라인 : HTTP 요청 메시지인 경우 요청 라인, HTTP 응답 메시지인 경우 상태 라인
- 요청 라인
	- 메서드 : 클라이언트가 서버의 자원에 대해 수행할 작업의 종류
		- GET, POST, PUT, DELETE
	- 요청 대상 : HTTP 요청을 보낼 서버의 자원
		- 보통 URL 명시 
		- `http://www.example.com/hello?q=world`인 경우 `/hello?q=world`가 요청 대상
		- 하위 경로 없어도 요청 대상은 `/`로 표기
	- HTTP 버전 : `HTTP/<버전>`의 형태 
- 상태 라인
	- 상태 코드 : 요청에 대한 결과를 판단하는 세 자리 정수
	- 이유 구문 : 상태 코드에 대한 문자열 형태의 설명
	- 예시
		- `HTTP/1.1 200 OK`
		- `HTTP/1.1 404 Not Found`

##### 필드 라인 : 0개 이상의 HTTP 헤더 명시
- HTTP 헤더 : 통신에 필요한 부가 정보
- 콜론(:)을 기준으로 헤더 이름과 하나 이상의 헤더 값으로 구성

##### 메시지 본문 : HTTP 요청 혹은 응답 메시지에서 본문이 필요한 경우 명시
- 존재하지 않을 수 있고 다양한 콘텐츠 타입이 사용될 수 있음

### HTTP 메서드
##### GET  : 자원을 습득하기 위한 메서드
- 웹 브라우저를 통해 조회하는 대부분의 자원은 GET 요청에 대한 응답
- 요청 메시지
	```
	GET /index.html?name1=value1&name2=value2 HTTP/1.1
	Host: www.example.com
	Accept: *
	```
	-  요청 대상 : 쿼리 문자열 많이 사용
	- Host 헤더 : 요청을 보낼 호스트 명시
- 응답 메시지
	```
	HTTP/1.1 200 OK
	Content-Type: text/html
	Content-Length: 1234
	
	<!DOCTYPE html>
	<html>
	<head>
		<title>Example Page</title>
	</head>
	<body>
		<h1>Hello,  World!</h1>
	</body>
	</html>
	```

##### HEAD : GET과 동일하나, 헤더만을 받는 메서드
- 응답 메시지에 본문 포함 안되는 것이 GET과의 차이
##### POST : 서버로 하여금 특정 작업을 처리하게끔 하는 메서드
- 범용성 매우 넓음
	- 많은 경우 클라이언트가 서버에 새로운 자원을 생성하고자 할 때 사용
- 요청 메시지 (게시글 작성)
	```
	POST /posting HTTP/1.1
	Host: example.com
	...
	
	{
		"Id": 1,
		"Title": "제목",
		"Contents": "내용"
	}
	```

- 응답 메시지
	```
	HTTP/1.1 201 Created
	Content-Type: application/json
	Content-Length: 100
	Date: Mon, 14 Oct 2024 16:35:00 PST
	Location: /posting/1
	
	{
		"Id": 1,
		"Title": "제목",
		"Contents": "내용"
	}
	```
	- Location 헤더로 새로 생성된 자원의 위치 전달

##### PUT : 자원을 대체하기 위한 메서드
- 요청 자원이 없다면 새롭게 생성, 있다면 대체
##### PATCH : 자원에 대한 부분적 수정을 위한 메서드
- PUT과 달리 부분적 수정
##### DELETE : 자원을 삭제하기 위한 메서드
- 요청 메시지
	```
	DELETE /texts/a.txt HTTP/1.1
	Host: example.com
	```

> [!info]+ API 문서
> 어떤 URL로 어떤 요청을 받았을 때 서버는 어떻게 응답할지 정리한 문서
> - 같은 URL에 대한 요청이라도 사용된 메서드가 다르면 각기 다른 요청
> - 따라서 이것을 설계하는 것은 개발자의 몫

### HTTP 상태 코드 : 요청에 대한 결과를 나타내는 세 자리 정수
##### 200번대: 성공 상태 코드
- 200 (OK) : 요청이 성공했음
- 201 (Created) : 요청이 성공했으며, 새로운 자원이 생성되었음
- 202 (Accepted) : 요청을 잘 받았으나, 아직 요청한 작업을 끝내지 않았음
	- 대용량 파일 업로드 작업, 배치 작업 등
- 204 (No Content) : 요청이 성공했지만, 메시지 본문으로 표시할 데이터가 없음
	- DELETE 요청 등

##### 300번대: 리다이렉션[^2] 상태 코드
- 동작 방식
	- 초기 요청 : `GET /old HTTP/1.1`
	- 응답 : `HTTP/1.1 301 Moved Pemanently \n Location: /new`
	- 재요청 : `GET /new HTTP/1.1`
	- 응답 : `HTTP/1.1 200 OK`

- 영구적 리다이렉션 : 자원이 완전히 새로운 곳으로 이동하여 경로가 영구적으로 재지정
	- 301 (Moved Permanently) : 영구적 리다이렉션; 재요청 메서드 변경될 수 있음
		- 처음에는 GET, 두번째 재요청은 POST가 될 수도 있음
	- 308 (Permanent Redirect) : 영구적 리다이렉션; 재요청 메서드 변경되지 않음
- 일시적 리다이렉션 : 자원의 위치 임시로 변경 혹은 임시로 사용할 URL이 필요한 경우에 사용
	- 302 (Found) : 일시적 리다이렉션; 재요청 메서드 변경될 수 있음
	- 303 (See Other) : 일시적 리다이렉션; 재요청 메서드 GET으로 변경
	- 307 (Temporary Redirect) : 일시적 리다이렉션; 재요청 메서드 변경되지 않음

- 요청을 다른 URL로 이동시킬 수도 있고 '캐시'로 이동시킬 수도 있음
	- 304 (Not Modified) 가 캐시 관련 상태 코드 ^befeea
	- [[15. HTTP 헤더와 HTTP 기반 기술#캐시 불필요한 대역폭 낭비와 응답 지연을 방지하기 위해 정보의 사본을 임시로 저장하는 기술|캐시]] 부분에서 설명

##### 400번대: 클라이언트 에러 상태 코드
- 400 (Bad Request) : 클라이언트의 요청이 잘못되었음
- 401 (Unauthorized) : 요청한 자원에 대한 유효한 인증이 없음 ^3109ad
	- 서버는 응답 메시지에 WWW-Authenticate 헤더를 통해 인증 방법을 알려줘야 함
- 403 (Forbidden) : 요청이 서버에 의해 거부됨
	- 접근 권한이 없을 경우
- 404 (Not Found) : 요청받은 자원을 찾을 수 없음
- 405 (Method Not Allowed) : 요청한 메서드를 지원하지 않음 ^3bd9a7

##### 500번대: 서버 에러 상태 코드
- 500 (Internal Server Error) : 요청을 처리할 수 없음
- 502 (Bad Gateway) : 중간 서버의 통신 오류
	- 보통 클라이언트와 서버 사이에는 게이트웨이를 비롯한 여러 중간 서버 존재
- 503 (Service Unavailable) : 현재는 요청을 처리할 수 없으나 추후 가능할 수도 있음 ^0a6cae
	- 서버 과부하 혹은 일시적인 점검 상태인 경우 발생

> [!info]+ HTTP의 발전
> 1. **HTTP/0.9**
> 	- 초창기 버전
> 	- GET만 사용 가능, 요청 메시지 한 줄로 구성
> 	- 헤더 미지원
> 2. **HTTP/1.0**
> 	- HEAD, POST 등 지원
> 	- 헤더 지원
> 	- 지속 연결 미지원
> 3. **HTTP/1.1**
> 	- 지속 연결 공식 지원
> 	- 파이프라이닝 지원 (특정 요청에 대한 응답 수신되기 전 다음 요청 보낼 수 있는 기능)
> 	- 콘텐츠 협상 기능 등 다양한 편의 기능 및 헤더 추가
> 4. **HTTP/2.0**
> 	- 1.1 버전 보완 및 개선
> 	- 송수신 효율 개선 : 헤더 압축 전송, 바이너리 데이터 기반 메시지 송수신
> 	- 서버 푸시 제공 (클라이언트가 요청안했더라도 필요할 것으로 예상되는 자원 미리 전송)
> 	- 멀티플렉싱[^3] 도입으로 HOL 블로킹[^4] 완화 
> 5. **HTTP/3.0**
> 	- 기존 HTTP는 모두 TCP기반인 반면 3.0은 UDP 기반으로 구현된 QUIC (Quick UDP Internet Connections) 프로토콜 기반으로 동작
> 	- 속도 측면에서 큰 개선



[^1]: Multipurpose Internet Mail Extensions Type

[^2]: 클라이언트가 요청한 자원이 다른 곳에 있을 때, 클라이언트의 요청을 다른 곳으로 이동시키는 것

[^3]: 여러 스트림을 이용해 병렬적으로 메시지 주고받는 기술

[^4]: Head-of-Line 블로킹. 같은 큐에 대기하며 순차적으로 처리되는 여러 패킷이 있을 때, 첫 번째 패킷의 처리 지연으로 나머지도 지연되는 문제
