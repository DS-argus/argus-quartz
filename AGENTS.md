# Blog Writing Agent Guide

이 저장소에서 블로그/노트 글 작성 요청을 받을 때 아래 규칙을 기본으로 따른다.

## Reference Files

- `content/Obsidian/Quartz/Obsidian 블로그 첨부파일 자동 정리.md`
- `content/Dev/python/uv - An extremely fast Python package and project manager.md`

## New Draft Workflow

- 사용자가 "새 글 작성"을 요청하면 초안 파일은 항상 `200. Inbox/201. Thinkbox`에 생성한다.
- 새 파일은 반드시 `999. ETC/Templates/Template_Inbox.md`를 복사해서 시작한다.
- 템플릿 복사 후 제목에 맞는 파일명으로 저장하고 내용을 작성한다.
- 템플릿 frontmatter(`tags`, `created`, `updated`) 구조는 유지한다.

## Language And Tone

- 기본 언어는 한국어로 작성한다.
- 설명형 문체를 사용한다.
- 불필요한 감탄/수사보다 실무 중심으로 작성한다.
- 개념 설명 후 바로 명령어/예시를 배치한다.

## Document Structure

- 문서 시작은 YAML frontmatter를 사용한다.
- 기본 frontmatter 필드:
  - `tags`
  - `created`
  - `updated`
- 본문은 `###` 단위의 큰 섹션으로 나눈다.
- 큰 섹션 전환 시 `---` 구분선을 사용한다.
- 필요하면 `####`, `#####`로 하위 구성을 만든다.

## Callout Rules

- 요약/핵심은 Obsidian callout을 적극 사용한다.
- 대표 패턴:
  - `> [!info]+ ...`
  - `> [!tip]+ ...`
  - `> [!note]+ ...`

AI가 생성한 글인 경우 문서 상단(frontmatter 바로 아래)에 아래 callout을 추가한다.

```md
> [!warning]+ Alert
> Generated by AI (Codex)
```

## Content Style

- 섹션 제목은 숫자 기반 순서형 구성을 선호한다. 예: `### 1. 설치`
- 명령어는 반드시 fenced code block으로 작성한다. 언어 표기는 `bash`, `sh`, `toml`, `json` 등 적절히 지정한다.
- 명령어 예시는 복붙 가능한 형태로 제공한다.
- 필요할 때만 이미지/내부 링크(`![[...]]`)를 사용한다.
- 긴 설명은 목록으로 쪼개 가독성을 높인다.

## Technical Writing Preferences

- "왜 필요한지"를 먼저 짧게 설명하고, "어떻게 하는지"를 단계형으로 작성한다.
- 도구 비교나 마이그레이션 글에서는 실무 커맨드를 우선 제시한다.
- 개념 설명 시 핵심 용어(예: resolve/install)는 구분해서 정리한다.
- 주의사항/제약사항은 별도 callout 또는 bullet로 분리한다.

## Consistency Checklist

글 작성 후 아래를 확인한다.

1. frontmatter(`tags`, `created`, `updated`)가 있는가
2. 섹션 구조가 `###` 중심으로 정리되었는가
3. callout이 목적에 맞게 사용되었는가
4. 명령어 블록이 실행 가능한 형태인가
5. 문체가 레퍼런스와 유사한가
